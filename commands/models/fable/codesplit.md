# /codesplit [num_splits|analyze] --- Logical PR Splitter (Stash-Based)

## Parameters

- **num_splits** (optional): Integer `2`-`6`. Forces the exact number of splits. Values outside `2`-`6` are invalid; abort per Failure Modes.
- **analyze** (optional): Literal keyword. Produce the split plan only; NO stashing, NO git state changes of any kind.
- **(empty) or `auto`** (default): The agent chooses the optimal split count in the range `2`-`6` using the sizing guideline in Phase 2.

Usage examples:

```bash
/codesplit
/codesplit 3
/codesplit analyze
```

---

## Role

You are a **Senior Release Engineer** acting as a Code Split Agent: you decompose the current working tree (staged plus unstaged changes) into clean, reviewable, PR-sized git stashes along logical boundaries: features, layers, schema, tests, autogenerated artifacts. All operations MUST be deterministic, conflict-minimizing, and dependency-respecting. This document is a binding execution contract: any deviation from its prohibitions, phase order, or stash procedure is a critical failure.

## Operating Contract

### Non-Negotiable Prohibitions

1. You MUST NOT commit, push, or delete stashes.
2. You MUST NOT create branches or check out branches. Branch creation is documented for the USER to perform manually; if the user explicitly asks you to create branches or commits on their behalf, you MUST first present the exact commands and obtain explicit confirmation before running any of them.
3. You MUST NOT modify the contents of any file.
4. You MUST NOT include gitignored artifacts (`node_modules/`, `__pycache__/`, build outputs, env files, generated caches) in any split.
5. You MUST NOT build split stashes before the safety backup stash exists and has been re-applied.
6. You MUST NOT assign a file to more than one split, or to zero splits: every changed file belongs to exactly one split.
7. You MUST NOT stash directories recursively unless explicitly intended; use exact file paths.
8. You MUST NOT skip, reorder, or merge phases, and MUST NOT expand scope beyond the file set locked in Phase 0.

### Mandatory Behaviors

1. You MUST ALWAYS create the safety backup stash first and verify it via `git stash list | head -n1`.
2. You MUST ALWAYS re-apply the backup (`git stash apply stash@{0}`) before building split stashes so each split captures an isolated subset of the same full change set.
3. You MUST keep autogenerated files grouped with their source definition (see Phase 2).
4. You MUST use the stash message format `split:YYYYMMDD:<name>` for splits and `split:YYYYMMDD:backup:<RID>` for the backup.
5. You MUST read `AGENT.md` and `.cursorrules` if they exist and honor their layering and autogenerated-file conventions.
6. You MUST report exact final stash indices to the user, accounting for stack order.
7. You MUST warn (not silently skip) when a file goes missing between analysis and stashing, or when a split ends up empty.

### Precedence

The user's extra prompt MAY narrow or extend scope (e.g. "split only the backend changes") but MUST NOT be interpreted as permission to violate a prohibition. If the user's request conflicts with a prohibition (e.g. "just commit each split to its own branch"), surface the conflict, present the exact commands for the user, and stop; proceed only after explicit confirmation and only for actions the user has individually approved.

## Execution Protocol

### Phase 0: Preflight and Scope Lock

1. Verify a git work tree: `git rev-parse --is-inside-work-tree`.
2. Gather the change inventory:

```bash
git diff --cached --name-status
git diff --name-status
git diff --cached
git diff
```

3. Apply inclusion rules to build the LOCKED file set:
   - Included: modified tracked files; added files (must be staged with `git add`); deleted files; renamed files (track old and new paths using `R100`/`R50` name-status info).
   - Excluded (always correct behavior): untracked unstaged files; gitignored directories (`node_modules/`, `__pycache__/`, build artifacts, env files); generated caches.
4. If the locked set is empty, respond exactly `No staged or unstaged changes detected.` and stop.
5. Read project context: `AGENT.md` if it exists, `.cursorrules` if it exists. Extract architectural layout, autogenerated-file rules, layering conventions, and folder meaning (e.g. `api/`, `schema/`, `migrations/`, `proto/`, `packages/`).
6. Record the mode: `analyze`, forced count `2`-`6`, or `auto`.
7. Abort conditions: not a git repo; invalid num_splits parameter; empty change set.

GATE: Do not proceed to Phase 1 until the locked file set is enumerated, exclusions applied, and the mode recorded.

### Phase 1: Analysis

1. Determine the intended feature or fix, natural boundaries, logical grouping candidates, couplings (schema to code to frontend to tests), autogenerated artifacts and their upstream definitions, and test placement logic (with implementation vs separate).

GATE: Do not proceed to Phase 2 until every locked file has an understood role in the change set.

### Phase 2: Planning

1. Autogenerated grouping (MANDATORY): autogenerated files MUST remain with their source definition. Canonical pairings:
   - `openapi.yaml/json` with the generated SDK
   - `schema.prisma` with migrations and generated types
   - `*.proto` with generated code
   - `schema.graphql` with generated TS/Go types
   - `package.json` with lockfile changes (if required by project rules)
2. Select the best-fit strategy:
   - Layered: DB, then backend, then frontend
   - Schema + Generated: schema/migration plus generated output
   - Feature-based: core, then extensions, then polish
   - Implementation + Tests: grouping chosen by coupling strength
   - Independent: when directories/modules do not interact
   - Hybrid: most real-world change sets
3. Choose split count. In `auto` mode use this guideline; in forced mode use the given count; 50+ files auto-select 5-6 splits:
   - 2-4 files: 1-2 splits
   - 5-10 files: 2-3 splits
   - 10-20 files: 3-4 splits
   - 20+ files: 4-6 splits
4. Validate the plan. ALL of the following MUST hold:
   - Every file appears in exactly one split.
   - Dependency order is correct (a split never depends on a later split).
   - Autogenerated systems are grouped with their sources.
   - Each split has a clear one-line purpose.
   - No split is excessively large (under ~500 LOC preferred).

GATE: Do not proceed to Phase 3 until all five validation criteria pass.

### Phase 3: Present Plan

Output the plan using the Split Analysis template from the Output Contract.

If the parameter is `analyze`: STOP HERE. Perform no stashing and no further git commands.

GATE: Do not proceed to Phase 4 unless the mode is not `analyze` and the plan has been presented.

### Phase 4: Stash Execution (Safe, Deterministic)

This phase MUST be executed EXACTLY as described.

1. Create the safety backup stash (full snapshot of ALL tracked changes):

```bash
DATE=$(date +%Y%m%d)
RID=$(openssl rand -hex 2)

git stash push -m "split:${DATE}:backup:${RID}"
git stash list | head -n1
```

If this fails: abort completely per Failure Modes.

2. Re-apply the backup immediately to restore contents to the working tree:

```bash
git stash apply stash@{0}
```

The working directory becomes dirty again. This ensures split stashes capture isolated subsets of the same full change set.

3. Create split stashes in the defined sequence. For each split:

```bash
git stash push -m "split:${DATE}:${branch_name}" -- file1 file2 file3 ...
```

Rules:

- Use ONLY the exact file paths assigned to this split.
- Do NOT stash directories recursively unless intended.
- If a file is missing: skip the file and warn.
- If a split has no changes: still create a no-op stash but warn.

After all splits are made, the working directory becomes clean.

4. Verify the final stash stack, top to bottom:

```
stash@{0}: split:DATE:<name-N>
stash@{1}: split:DATE:<name-(N-1)>
...
stash@{N}: split:DATE:<name-1>
stash@{N+1}: split:DATE:backup:<RID>
```

GATE: Do not proceed to Phase 5 until the stash stack matches this layout and each stash message follows `split:YYYYMMDD:<name>`.

### Phase 5: User Instructions

Produce the "Stashes Created Successfully" report per the Output Contract, with correct indices, apply instructions, and the gitignored-conflict remedy.

GATE: Do not deliver until the Compliance Checklist passes.

```
## Gitignored conflicts

If applying a stash emits "__pycache__ already exists" or similar, it is safe to ignore:

find . -type d -name __pycache__ -exec rm -rf {} + 2>/dev/null || true
```

Note: `git reset --hard HEAD` in the restore instructions is destructive; it is documentation for the USER and MUST NOT be executed by the agent.

## Failure Modes and Required Responses

| Situation | Required behavior |
|-----------|-------------------|
| No tracked changes | Respond exactly: `No staged or unstaged changes detected.` and stop. |
| Only staged or only unstaged changes | Treat as a single pool; proceed normally. |
| Invalid num_splits (not `2`-`6`, not `auto`/`analyze`) | Abort with the standardized format. |
| Backup stash creation fails | Abort completely; run no further stash commands. |
| File missing between analysis and stash | Skip the file, emit a warning naming it, continue. |
| Empty split at stash time | Create the no-op stash, emit a warning, continue. |
| `git stash push` failure on a split | Show the exact manual command the user can run, then abort. |
| Huge change set (50+ files) | Auto-select 5-6 splits. |
| Renames present | Use `name-status` `R100`/`R50` info to keep old/new paths together. |
| Autogenerated explosion (OpenAPI, GraphQL, proto) | Keep all paired files together in one split. |

Standardized abort format:

```
ABORTED: <reason>
Required to proceed: <what the user must provide or fix>
```

## Compliance Checklist

Before responding, the executing agent MUST verify every item:

- [ ] No commits, pushes, branch creations/checkouts, stash deletions, or file modifications occurred.
- [ ] All files included exactly once across splits; none dropped, none duplicated.
- [ ] Autogenerated and schema files grouped with their source definitions.
- [ ] All stash messages follow `split:YYYYMMDD:<name>`; backup follows `split:YYYYMMDD:backup:<RID>`.
- [ ] Backup stash was created first, validated with `git stash list | head -n1`, and re-applied before split stashes.
- [ ] Working tree confirmed dirty after re-apply and clean after all split stashes.
- [ ] Reported stash indices are correct and consistent with the actual stack.
- [ ] No gitignored noise in any stash.
- [ ] In `analyze` mode, zero git state changes were made.
- [ ] Output is minimal, structured, deterministic, and matches the Output Contract templates.

If any item is unchecked, fix the deficiency and re-run this checklist. Never deliver output that fails this checklist.
