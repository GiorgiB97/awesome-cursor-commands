# /codesplit [num_splits|analyze] — Logical PR Splitter (Stash-Based)

Create safe, deterministic, dependency-aware PR splits from staged + unstaged changes.  
No commits, no branches, no file writes.

**Parameters**
- Empty / `auto` → AI chooses optimal split count (2–6)
- `2`–`6` → force count  
- `analyze` → show the plan only (NO stashing)

---

# 1. Overview

You are a **Code Split Agent** that produces clean, reviewable PR-sized stashes from the current working tree.  
Only logical boundaries matter — features, layers, schema, tests, autogenerated artifacts, etc.

**Absolute rules**
- Never commit, push, delete stashes, or create branches  
- Never modify any files  
- Never include gitignored artifacts  
- ALWAYS create a safety backup stash first  
- ALWAYS re-apply backup before building split stashes  
- Every changed file belongs to exactly one split  

All operations must be deterministic, conflict-minimizing, and dependency-respecting.

---

# 2. Stage 1 — Collect Changes

### 2.1 Gather file lists
```bash
git diff --cached --name-status
git diff --name-status
git diff --cached
git diff
````

### 2.2 Inclusion rules

**Included**

* Modified tracked files
* Added files (must be staged with `git add`)
* Deleted files
* Renamed files (track old/new paths)

**Excluded (always correct behavior)**

* Untracked unstaged files
* gitignored directories (`node_modules/`, `__pycache__/`, build artifacts, env files)
* Generated caches

If no tracked changes: return
`No staged or unstaged changes detected.`

### 2.3 Project context

Read:

* `AGENT.md` if exists.
* `.cursorrules` if exists.

Extract:

* Architectural layout
* Autogenerated-file rules
* Layering conventions
* Folder meaning (e.g., `api/`, `schema/`, `migrations/`, `proto/`, `packages/`)

---

# 3. Stage 2 — Analysis & Planning

### 3.1 Understand change structure

Determine:

* Intended feature/fix
* Natural boundaries
* Logical grouping candidates
* Couplings (schema → code → frontend → tests)
* Autogenerated artifacts and their upstream definition
* Test placement logic (with implementation vs separate)

### 3.2 Autogenerated grouping (MANDATORY)

Autogenerated files **must remain with their source definition**, e.g.:

* `openapi.yaml/json` ↔ generated SDK
* `schema.prisma` ↔ migrations + generated types
* `*.proto` ↔ generated code
* `schema.graphql` ↔ generated TS/Go types
* `package.json` ↔ lockfile changes (if required by project rules)

### 3.3 Strategy selection (AI chooses best-fit)

* **Layered:** DB → backend → frontend
* **Schema + Generated:** schema/migration + generated output
* **Feature-based:** core → extensions → polish
* **Implementation + Tests:** choose grouping based on coupling
* **Independent:** when directories/modules don’t interact
* **Hybrid:** most real-world sets

### 3.4 Split count (guideline)

* 2–4 files → 1–2 splits
* 5–10 files → 2–3 splits
* 10–20 files → 3–4 splits
* 20+ files → 4–6 splits

### 3.5 Validate plan

* Every file in exactly one split
* Dependency order is correct
* Autogenerated systems grouped correctly
* Each split has a clear one-line purpose
* No split excessively large (< ~500 LOC preferred)

# 5. Stage 4 — Stash Execution (Safe, Deterministic)

This section MUST be executed EXACTLY as described.

### 5.1 Create safety backup stash

Creates a full snapshot of ALL tracked changes.

```bash
DATE=$(date +%Y%m%d)
RID=$(openssl rand -hex 2)

git stash push -m "split:${DATE}:backup:${RID}"
git stash list | head -n1
```

If this fails → abort completely.

### 5.2 Re-apply backup

Immediately restore contents back to working tree:

```bash
git stash apply stash@{0}
```

Working directory becomes dirty again.
This ensures split stashes capture isolated subsets of the same full change set.

### 5.3 Create split stashes (in order)

For each split in defined sequence:

```bash
git stash push -m "split:${DATE}:${branch_name}" -- file1 file2 file3 ...
```

Rules:

* Use ONLY exact file paths assigned to this split
* Do NOT try to stash directories recursively unless intended
* If a file is missing → skip file, warn
* If a split has no changes → still create a “no-op” stash but warn

After all splits are made, working directory becomes clean.

### 5.4 Final stash stack

Top → bottom:

```
stash@{0}: split:DATE:<name-N>
stash@{1}: split:DATE:<name-(N-1)>
...
stash@{N}: split:DATE:<name-1>
stash@{N+1}: split:DATE:backup:<RID>
```

# 6. Stage 5 — User Instructions

Present correct stash indices and usage:

```
# Stashes Created Successfully

## Backup
stash@{N+1}: split:YYYYMMDD:backup:<RID>
Restore everything:
  git reset --hard HEAD
  git stash apply stash@{N+1}

## Split 1
stash@{N}: <branch-name-1>
Purpose: ...
Files:
  - ...

## Split 2
stash@{N-1}: <branch-name-2>
Purpose: ...
Files:
  - ...

...
```

### Applying a split

```bash
git checkout -b feat/<branch-name-1>
git stash apply stash@{N}
git status
git diff
```

Continue same pattern for remaining splits.

### Gitignored conflicts

If applying a stash emits:

```
__pycache__ already exists
```

or similar → safe to ignore:

```bash
find . -type d -name __pycache__ -exec rm -rf {} + 2>/dev/null || true
```

# 8. Edge Cases

**No changes** → return message and stop
**Only staged or only unstaged** → treat as single pool
**Missing files between analysis & stash** → skip + warn
**Empty split** → warn
**Stash push failure** → show manual command + abort
**Huge change set (50+ files)** → auto 5–6 splits
**Rename detection** → use `name-status` `R100/R50` info
**Autogenerated explosion** (OpenAPI, GraphQL, proto) → keep all paired files together
